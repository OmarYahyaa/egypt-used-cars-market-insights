# Raw Table Design v1

**Project:** Egypt Used Cars Market Analysis  
**Document:** `03_raw_table_design.md`

## 1. Purpose

This document defines the first PostgreSQL raw table design for the Egypt Used Cars SQL project.

The raw table is a safe landing layer for the original CSV data. Its purpose is to preserve the source values exactly as they came from the file, while adding minimal metadata columns for traceability and future data engineering work.

## 2. Table Name

```sql
raw.raw_used_car_listings_aug_2025
```

### Reasoning

- The schema `raw` makes it clear that this table belongs to the raw/landing layer.
- `used_car_listings` describes the entity being stored.
- `aug_2025` identifies the dataset snapshot/source period.

## 3. Source Columns

The table includes all 12 columns from the original CSV:

1. `title`
2. `company`
3. `model`
4. `year`
5. `price`
6. `mileage`
7. `color`
8. `transmission`
9. `location`
10. `date_posted`
11. `features`
12. `detail_link`

## 4. Metadata Columns

The table also includes metadata columns created by the database/import process.

### 4.1 `raw_listing_id`

- Generated by PostgreSQL.
- Used as the internal primary key for the raw table.
- Recommended data type:

```sql
BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY
```

### 4.2 `source_file_name`

- Stores the name of the CSV file loaded into the database.
- Example:

```text
hatla2ee_cars_august_2025.csv
```

- Useful for future comparisons if more files are added later.

### 4.3 `loaded_at`

- Stores the timestamp when the row was loaded into PostgreSQL.
- Recommended data type:

```sql
TIMESTAMPTZ DEFAULT NOW()
```

- Useful for auditing and debugging.

### Optional Future Metadata Column

#### `source_row_number`

- Can be added later if a more advanced import process is built.
- Useful for tracing a database row back to the exact row in the original CSV.

## 5. Primary Key

```sql
raw_listing_id
```

### Reasoning

- The CSV does not provide a clean, reliable primary key.
- `detail_link` is useful for traceability and duplicate checks, but it is not fully unique.
- A generated internal key is safer than forcing a messy source column to act as a primary key.

## 6. Data Type Strategy

In the raw table, all source columns should be stored as `TEXT`:

```sql
title TEXT
company TEXT
model TEXT
year TEXT
price TEXT
mileage TEXT
color TEXT
transmission TEXT
location TEXT
date_posted TEXT
features TEXT
detail_link TEXT
```

### Reasoning

- `price` contains commas and currency text, such as `1,640,000 EGP`.
- `mileage` contains commas and unit text, such as `58,000 Km`.
- `year` contains invalid values such as `1100` and `5008`.
- `date_posted` should preserve the raw source value before conversion.
- `features` is a pipe-separated text list.
- `transmission` contains mostly `N/A` values and has very low coverage.
- `location` contains mixed and possibly incorrect values.

The cleaned layer will later convert selected columns into proper analytical types, such as:

- `price_egp NUMERIC`
- `mileage_km NUMERIC`
- `manufacturing_year SMALLINT`
- `date_posted DATE`

## 7. Why This Design Is Safe

This design is safe because:

- It preserves the raw source data without manual editing.
- It avoids import failures caused by messy numeric/date fields.
- It separates raw storage from cleaning and analysis.
- It keeps `detail_link` for traceability and duplicate checks.
- It adds metadata columns for future auditing, reloading, and scraping upgrades.
- It supports a clear data pipeline structure:

```text
raw data -> clean data -> analytical queries
```

## 8. Important Design Decisions

- Do not clean data during import.
- Do not manually fix the CSV in Excel.
- Do not drop `detail_link` from the raw layer.
- Do not store `year` as `SMALLINT` in the raw layer.
- Do not store `price` or `mileage` as numeric in the raw layer.
- Do not store `date_posted` as `DATE` in the raw layer.
- Generate `raw_listing_id` inside PostgreSQL.
- Use `source_file_name` and `loaded_at` to support traceability.

## 9. Related Rulebook Principle

### Principle

Raw table columns should preserve source values first, even when the values look numeric or date-like.

### Why This Matters

If a column contains invalid, ambiguous, or messy values, forcing it into a strict type during raw import can hide data quality problems or cause import failure.

### How It Appeared in the Project

The `year` column looked numeric, but it contained invalid values like `1100` and `5008`. So `year` should be stored as `TEXT` in the raw table and converted only later in the cleaned layer.
